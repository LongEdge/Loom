# Loom v0.1 建议

## 1. 文档目的

用于沉淀你与架构师沟通后的建议项，作为 v0.1 阶段架构与实现协作的参考输入。

## 2. 适用范围

- 适用于 `DOCUMENTS/v0.1` 范围内的架构、模块边界、实现落地相关决策。
- 面向 Loom v0.1 阶段的协作成员。

## 3. 规则声明

本文档中的建议为**推荐实践**，**不作为硬性约束**；但在不与现实约束冲突的前提下，**最好遵守**。

若因实际约束无法遵守某条建议，应在对应文档中补充简要说明（原因、影响范围、替代方案）。

## 4. 建议清单

### 建议 1：FastAPI 可作为交互层实现，但不进入内核层

- 建议内容：Web 后端使用 FastAPI 是可行的，但 FastAPI 仅用于协议桥接，不用于承载内核业务规则。
- 适用场景：设计或实现 HTTP API、鉴权、请求解析、响应封装。
- 不遵守时说明要求：如需在非交互层引入 FastAPI 相关能力，需说明必要性、边界及替代不可行原因。

### 建议 2：内核层禁止直接依赖框架与 I/O 技术细节

- 建议内容：内核层（尤其 `domain`/`application`）避免直接引入 `fastapi`、数据库驱动、HTTP 客户端、外部网络调用等技术细节依赖。
- 适用场景：定义领域实体、用例编排、状态机、核心策略时。
- 不遵守时说明要求：需明确该依赖无法外置到 `ports/adapters` 的理由与影响范围。

### 建议 3：统一输入契约，先做输入归一再执行业务

- 建议内容：CLI 参数与 Web JSON 在进入内核前统一转换为应用层 DTO（Command/Query），避免内核感知入口差异。
- 适用场景：提交任务、查询状态、注册资产等跨入口用例。
- 不遵守时说明要求：需说明该用例为何不适合 DTO 归一，以及如何保证多入口一致性。

### 建议 4：统一接口以 Use Case 为核心，而非以路由或命令函数为核心

- 建议内容：内核统一接口应落在 `application/commands` 与 `application/queries`（或等价用例服务）上，交互层只负责调用。
- 适用场景：新增业务动作或重构已有入口时。
- 不遵守时说明要求：需说明为何不能抽象为稳定用例接口，以及后续维护策略。

### 建议 5：通过 Inbound/Outbound Ports 隔离业务与外部系统

- 建议内容：内核仅依赖端口契约；数据库、消息队列、执行器、日志系统等由适配器实现并注入。
- 适用场景：持久化、调度执行、日志写入、外部服务接入。
- 不遵守时说明要求：需说明直连外部实现的短期收益与长期风险，并标注修复计划。

### 建议 6：CLI 与 Web 采用同一调用链路

- 建议内容：推荐统一流转为“入口解析 -> DTO -> Use Case -> 纯结果 -> 入口层输出渲染”，保证行为一致。
- 适用场景：同一业务能力同时暴露给 CLI 与 Web 时。
- 不遵守时说明要求：需说明差异化路径的业务必要性，并给出一致性保障措施。

### 建议 7：依赖注入在外层完成组装，内核不承担装配职责

- 建议内容：在 FastAPI/Typer 启动阶段完成 Repository、Use Case、Adapter 的装配与注入；内核聚焦业务执行。
- 适用场景：应用启动、测试环境替身注入、不同部署形态切换。
- 不遵守时说明要求：需说明为何内核需要装配职责，以及对可测试性和替换性的影响。

### 建议 8：坚持 Thin Controller/Thin Command Handler

- 建议内容：Router/CLI Handler 仅做协议解析、参数校验、调用用例、结果映射，不写核心业务 if/else。
- 适用场景：新增接口、改造遗留接口、错误处理标准化。
- 不遵守时说明要求：需指出保留业务逻辑在入口层的范围与回收计划。

### 建议 9：优先构建可脱离框架运行的内核测试

- 建议内容：核心调度、状态迁移、规则校验优先做纯 Python 单元测试，必要时用内存仓储替身代替真实基础设施。
- 适用场景：内核规则新增、回归验证、重构风险控制。
- 不遵守时说明要求：需说明无法进行纯内核测试的阻碍点及补救测试方案。

### 建议 10：中介者/消息总线在 v0.1 定位为可选演进 PoC

- 建议内容：可通过轻量 PoC 验证 `bus.dispatch(command)` 的统一调度路径，但不作为 v0.1 强制落地项。
- 适用场景：团队希望统一入口调度模型、为后续分布式演进预留机制时。
- 不遵守时说明要求：无需阻塞交付；若采用该模式，需说明边界、注册机制与退化方案。
