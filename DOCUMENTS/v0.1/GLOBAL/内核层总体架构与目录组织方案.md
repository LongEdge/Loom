# Loom v0.1 内核层总体架构与目录组织方案

## 1. 目标与边界

本方案只覆盖 Loom 的内核层（`core/`），用于统一 `v0.1` 阶段的目录组织、分层职责和依赖方向。

内核层目标：

- 保持领域模型稳定，支撑后续 v0.2/v1.0 扩展。
- 把“业务规则”和“技术实现细节”解耦。
- 保证 CLI/Controller 等上层入口复用同一套内核能力。

不在本方案范围：

- 交互层细节（CLI 命令参数、HTTP 路由细节）。
- Agent 相关实现（已明确不在 v0.1 范围内）。

---

## 2. 分层定义（内核层）

内核层采用 6 层结构，自内向外依赖：

1. `shared`（共享内核）
2. `domain`（领域层）
3. `policy`（策略层）
4. `application`（应用编排层）
5. `ports`（端口契约层）
6. `adapters`（适配器实现层）

职责说明：

- `shared`：通用类型、错误码、时间/ID 抽象、基础工具。
- `domain`：实体、值对象、领域服务、领域事件，不依赖数据库或框架。
- `policy`：可替换的核心策略（调度、状态迁移、资源匹配规则）。
- `application`：用例编排（提交任务、分配资源、写入结果），组织事务边界。
- `ports`：内核对外部能力的抽象接口（仓储、消息、执行器、日志写入等）。
- `adapters`：对 `ports` 的具体实现（PostgreSQL、SQLite、本地执行、文件日志等）。

## 3. 目录组织方案（建议）

```text
loom/                                           # 项目根目录
  core/                                         # 内核层根目录，承载全部核心业务能力
    shared/                                     # 跨领域共享基础能力
      errors/                                   # 统一错误类型、错误码与异常映射
      types/                                    # 通用类型定义（ID、时间、分页、通用枚举等）
      utils/                                    # 与业务无关的基础工具函数
    domain/                                     # 领域模型层（不依赖框架/数据库）
      assets/                                   # 模块1：资产域（Machine/GPU/Dataset/Document/Artifact）
        entities/                               # 资产域实体与聚合根
        value_objects/                          # 资产域值对象（不可变语义对象）
        services/                               # 资产域领域服务与不变量规则
        events/                                 # 资产域事件（资产注册、变更、归档等）
      execution/                                # 模块2：执行域（Task/Run/Queue/Allocation）
        entities/                               # 执行域实体与状态模型
        value_objects/                          # 执行域值对象（资源请求、执行参数等）
        services/                               # 执行域领域服务（调度前后核心规则）
        events/                                 # 执行域事件（任务提交、启动、完成、失败等）
      orchestration/                            # 模块3：编排域（Experiment/Group/Batch/Sweep）
        entities/                               # 编排域实体（课题组/实验组/任务组等）
        value_objects/                          # 编排域值对象（编排参数、分组键、模板引用）
        services/                               # 编排域领域服务（实验上下文组织与约束）
        events/                                 # 编排域事件（实验创建、分组变更、批次触发）
      traceability/                             # 模块4：追溯域（输入-运行-输出关联）
        entities/                               # 追溯域实体（链路节点、结果对象）
        value_objects/                          # 追溯域值对象（指标快照、引用关系）
        services/                               # 追溯域领域服务（结果反查与闭环规则）
        events/                                 # 追溯域事件（结果登记、追溯链更新）
      interaction_context/                      # 模块5：交互上下文域（用户请求语义与调用意图）
        entities/                               # 交互上下文实体（调用会话、请求上下文）
        value_objects/                          # 交互上下文值对象（过滤条件、查询视图、命令参数）
        services/                               # 交互上下文领域服务（交互语义校验与意图归一）
        events/                                 # 交互上下文事件（调用发起、回执、审计触发）
    policy/                                     # 策略层（可替换算法与策略规则）
      scheduler/                                # 调度策略（FIFO/优先级/公平性扩展点）
      lifecycle/                                # 生命周期策略（状态迁移与约束）
      matching/                                 # 资源匹配策略（任务需求与资源能力匹配）
    application/                                # 应用层（用例编排与事务边界）
      commands/                                 # 写用例（提交任务、注册资产、记录结果）
      queries/                                  # 读用例（查询状态、检索结果、追溯查询）
      services/                                 # 应用服务（组合领域对象与端口调用）
      dto/                                      # 应用层输入/输出 DTO 定义
    ports/                                      # 端口层（内核对外部能力的抽象接口）
      inbound/                                  # 入站端口（供 CLI/Controller 调用的用例接口）
      outbound/                                 # 出站端口（仓储、执行器、日志等外部接口）
    adapters/                                   # 适配器层（端口的具体技术实现）
      persistence/                              # 持久化适配器
        postgres/                               # PostgreSQL 实现（仓储/查询实现）
        sqlite/                                 # SQLite 实现（本地开发/轻量部署）
      scheduler/                                # 调度相关适配实现（策略装配、执行桥接）
      runtime/                                  # 运行时适配实现（任务拉起、进程管理、回传）
      logging/                                  # 日志与观测适配实现（stdout/stderr/事件日志）
```

设计说明：

- `domain/assets`：资产统一建模与索引语义，关注“有什么资产、资产属性是什么”。
- `domain/execution`：任务执行生命周期，关注“任务如何被调度和运行”。
- `domain/orchestration`：实验组织语义，关注“任务为何执行、属于哪组实验”。
- `domain/traceability`：结果闭环语义，关注“结果如何反查输入与运行上下文”。
- `domain/interaction_context`：交互语义收敛，关注“外部请求如何映射为稳定领域意图”。
- `application/commands`：写操作用例（提交任务、注册资产、记录运行结果）。
- `application/queries`：读操作用例（状态查询、结果检索、资产关系查询）。
- `ports/outbound`：数据库、执行器、日志系统等外部依赖抽象。
- `adapters/*`：各类外部依赖实现，严格通过 `ports` 连接内核。

---

## 4. 依赖规则（必须遵守）

允许依赖方向：

- `shared <- domain <- policy <- application`
- `application -> ports`
- `adapters -> ports`
- `adapters` 可依赖 `shared/domain`，但不得反向依赖 `application`

禁止项：

- `domain` 直接依赖 ORM/HTTP/CLI 框架
- `application` 直接依赖具体数据库驱动
- `ports` 引用 `adapters` 中实现类型

判定原则：

- 只要出现“可被替换的外部系统”，必须经 `ports` 抽象。
- 只要是“业务真理/业务不变量”，必须沉到 `domain` 或 `policy`。

模块边界规则（为未来微服务拆分预留）：

- `domain` 五个模块按业务语义独立归档，不允许把某模块实体混放到其他模块目录。
- 跨模块协作优先通过 `events` 或 `application` 层编排，不直接耦合对方内部实现。
- `policy/application/ports/adapters` 保持现有顶层结构，但其实现必须按五模块边界组织引用。
