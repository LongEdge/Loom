# 技术选型



## 存储方案

**PostgreSQL + 本地/NAS 共享存储 + 路径索引**
 优点是部署最简单，和实验室现实环境最贴近，不需要先搭完整对象存储体系；数据集、checkpoint、日志都可以继续放在已有目录里，Loom 只登记 `uri`、checksum、owner、version、task_id、run_id。缺点是后期做跨机器复制、生命周期管理、权限隔离会弱一些，但这不是 v0.1 最优先的问题。



我们承诺在v0.2当中添加s3协议接口，并且支持同步功能



## Web端搭建

### 后端搭建

后端:FastAPI

ORM：SQLAIchemy2.0 + AIembic

数据校验和 API schema 用 Pydantic v2

搜索先不要上 ES / OpenSearch。v0.1 的资产规模不大，**PostgreSQL + trigram/full-text + tags** 足够。

### 前端搭建

**核心框架**：React 18 + TypeScript + Vite。Vite 提供极速的冷启动和热重载，TypeScript 保证与后端 Pydantic Schema 映射时的类型安全，降低前后端联调的沟通成本。

**UI 组件库**：Ant Design (AntD) 或 Radix UI + Tailwind CSS。由于 Loom 偏向于后台管理、资产表格展示与复杂表单交互，AntD 提供的高级表格（支持筛选、排序、树状展开）能极大提升研发效率。

**状态管理与数据请求**：React Query (TanStack Query) + Axios。替代传统的 Redux，专门用于管理服务端状态，天然支持数据的缓存、后台轮询（适合展示实验的 Running 状态）和请求重试。

**可视化图表（预留）**：ECharts 或 Apache ECharts for React。用于在 v0.1 阶段展示轻量级的资源占用折线图或实验 Metric 曲线。

## CLI搭建

CLI：Typer

**交互与格式化增强**：结合 `Rich` 库。为终端输出提供高亮的表格化资产列表、进度条（展示数据校验或打包进度）以及带颜色的日志输出，提升研究人员的终端使用体验。

**本地上下文管理**：利用本地隐藏目录（如 `~/.loom/config.yaml` 或 `~/.loom/credentials`）管理当前登录的用户 Token 和默认连接的 Workspace，实现类似 `git` 或 `kubectl` 的丝滑命令体验。
